<p><b>Node.js</b> is an <a href="/wiki/Open-source_software" title="Open-source software">open-source</a>, <a href="/wiki/Cross-platform" class="mw-redirect" title="Cross-platform">cross-platform</a>, <a href="/wiki/Front_end_and_back_end" class="mw-redirect" title="Front end and back end">back-end</a> <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> <a href="/wiki/Runtime_system" title="Runtime system">runtime environment</a> that runs on the <a href="/wiki/V8_(JavaScript_engine)" title="V8 (JavaScript engine)">V8 engine</a> and executes JavaScript code outside a <a href="/wiki/Web_browser" title="Web browser">web browser</a>. Node.js lets developers use JavaScript to write command line tools and for <a href="/wiki/Server-side_scripting" title="Server-side scripting">server-side scripting</a>â€”running scripts server-side to produce <a href="/wiki/Dynamic_web_page" title="Dynamic web page">dynamic web page</a> content before the page is sent to the user's web browser. Consequently, Node.js represents a "JavaScript everywhere" paradigm,<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">&#91;6&#93;</a></sup> unifying <a href="/wiki/Web_application" title="Web application">web-application</a> development around a single programming language, rather than different languages for server-side and client-side scripts.
</p><p>Node.js has an <a href="/wiki/Event-driven_architecture" title="Event-driven architecture">event-driven architecture</a> capable of <a href="/wiki/Asynchronous_I/O" title="Asynchronous I/O">asynchronous I/O</a>. These design choices aim to optimize <a href="/wiki/Throughput" title="Throughput">throughput</a> and <a href="/wiki/Scalability" title="Scalability">scalability</a> in web applications with many input/output operations, as well as for <a href="/wiki/Real-time_Web" class="mw-redirect" title="Real-time Web">real-time Web</a> applications (e.g., <a href="/wiki/Real-time_communication" title="Real-time communication">real-time communication</a> programs and <a href="/wiki/Browser_game" title="Browser game">browser games</a>).<sup id="cite_ref-readwrite_7-0" class="reference"><a href="#cite_note-readwrite-7">&#91;7&#93;</a></sup>
</p><p>The Node.js <a href="/wiki/Distributed_development" title="Distributed development">distributed development</a> project was previously governed by the Node.js Foundation,<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">&#91;8&#93;</a></sup> and has now merged with the <a href="/wiki/JS_Foundation" class="mw-redirect" title="JS Foundation">JS Foundation</a> to form the <a href="/wiki/OpenJS_Foundation" title="OpenJS Foundation">OpenJS Foundation</a>, which is facilitated by the <a href="/wiki/Linux_Foundation" title="Linux Foundation">Linux Foundation</a>'s Collaborative Projects program.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">&#91;9&#93;</a></sup>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright"><div class="thumbinner" style="width:222px;"><a href="/wiki/File:Ryan_Dahl.jpg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/b/b2/Ryan_Dahl.jpg/220px-Ryan_Dahl.jpg" decoding="async" width="220" height="330" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/b/b2/Ryan_Dahl.jpg/330px-Ryan_Dahl.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/b/b2/Ryan_Dahl.jpg/440px-Ryan_Dahl.jpg 2x" data-file-width="3456" data-file-height="5184" /></a>  <div class="thumbcaption"><div class="magnify"><a href="/wiki/File:Ryan_Dahl.jpg" class="internal" title="Enlarge"></a></div>Ryan Dahl, creator of Node.js, in 2010</div></div></div>
<p>Node.js was written initially by <a href="/wiki/Ryan_Dahl" title="Ryan Dahl">Ryan Dahl</a> in 2009,<sup id="cite_ref-training.com_24-0" class="reference"><a href="#cite_note-training.com-24">&#91;24&#93;</a></sup> about thirteen years after the introduction of the first server-side JavaScript environment, <a href="/wiki/Netscape" title="Netscape">Netscape's</a> LiveWire Pro Web.<sup id="cite_ref-25" class="reference"><a href="#cite_note-25">&#91;25&#93;</a></sup> The initial release supported only Linux and Mac OS X. Its development and maintenance was led by Dahl and later sponsored by <a href="/wiki/Joyent" title="Joyent">Joyent</a>.<sup id="cite_ref-Google_Groups_post_by_Ryan_Dahl_about_Joyent_26-0" class="reference"><a href="#cite_note-Google_Groups_post_by_Ryan_Dahl_about_Joyent-26">&#91;26&#93;</a></sup>
</p><p>Dahl criticized the limited possibilities of the most popular web server in 2009, <a href="/wiki/Apache_HTTP_Server" title="Apache HTTP Server">Apache HTTP Server</a>, to handle a lot of concurrent connections (up to 10,000 and more) and the most common way of creating code (sequential programming), when code either blocked the entire process or implied multiple execution stacks in the case of simultaneous connections.<sup id="cite_ref-27" class="reference"><a href="#cite_note-27">&#91;27&#93;</a></sup>
</p><p>Dahl demonstrated the project at the inaugural European JSConf on November 8, 2009.<sup id="cite_ref-b3_28-0" class="reference"><a href="#cite_note-b3-28">&#91;28&#93;</a></sup><sup id="cite_ref-29" class="reference"><a href="#cite_note-29">&#91;29&#93;</a></sup><sup id="cite_ref-30" class="reference"><a href="#cite_note-30">&#91;30&#93;</a></sup> Node.js combined <a href="/wiki/Google" title="Google">Google</a>'s <a href="/wiki/Chrome_V8" class="mw-redirect" title="Chrome V8">V8</a> JavaScript engine, an <a href="/wiki/Event_loop" title="Event loop">event loop</a>, and a low-level <a href="/wiki/Input/output" title="Input/output">I/O</a> <a href="/wiki/Application_programming_interface" class="mw-redirect" title="Application programming interface">API</a>.<sup id="cite_ref-b1_31-0" class="reference"><a href="#cite_note-b1-31">&#91;31&#93;</a></sup>
</p><p>In January 2010, a <a href="/wiki/Package_manager" title="Package manager">package manager</a> was introduced for the Node.js environment called <a href="/wiki/Npm_(software)" title="Npm (software)"><i>npm</i></a>.<sup id="cite_ref-32" class="reference"><a href="#cite_note-32">&#91;32&#93;</a></sup> The package manager makes it easier for programmers to publish and share source code of Node.js packages and is designed to simplify installation, updating, and uninstallation of packages.<sup id="cite_ref-b1_31-1" class="reference"><a href="#cite_note-b1-31">&#91;31&#93;</a></sup>
</p><p>In June 2011, Microsoft and Joyent implemented a native <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Windows</a> version of Node.js.<sup id="cite_ref-33" class="reference"><a href="#cite_note-33">&#91;33&#93;</a></sup> The first Node.js build supporting Windows was released in July 2011.
</p><p>In January 2012, Dahl stepped aside, promoting coworker and <i><a href="/wiki/Npm_(software)" title="Npm (software)">npm</a></i> creator Isaac Schlueter to manage the project.<sup id="cite_ref-34" class="reference"><a href="#cite_note-34">&#91;34&#93;</a></sup> In January 2014, Schlueter announced that Timothy J. Fontaine would lead the project.<sup id="cite_ref-35" class="reference"><a href="#cite_note-35">&#91;35&#93;</a></sup>
</p><p>In December 2014, Fedor Indutny started io.js, a <a href="/wiki/Fork_(software_development)" title="Fork (software development)">fork</a> of Node.js. Due to the internal conflict over Joyent's governance, io.js was created as an <a href="/wiki/Open_governance" class="mw-redirect" title="Open governance">open governance</a> alternative with a separate technical committee.<sup id="cite_ref-Krill,_Paul_36-0" class="reference"><a href="#cite_note-Krill,_Paul-36">&#91;36&#93;</a></sup><sup id="cite_ref-iwtw_37-0" class="reference"><a href="#cite_note-iwtw-37">&#91;37&#93;</a></sup> Unlike Node.js,<sup id="cite_ref-38" class="reference"><a href="#cite_note-38">&#91;38&#93;</a></sup> the authors planned to keep io.js up-to-date with the latest releases of the Google V8 JavaScript engine.<sup id="cite_ref-Mikeal,_Rogers_39-0" class="reference"><a href="#cite_note-Mikeal,_Rogers-39">&#91;39&#93;</a></sup>
</p><p>In February 2015, the intent to form a neutral Node.js Foundation was announced. By June 2015, the Node.js and io.js communities voted to work together under the Node.js Foundation.<sup id="cite_ref-40" class="reference"><a href="#cite_note-40">&#91;40&#93;</a></sup>
</p><p>In September 2015, Node.js v0.12 and io.js v3.3 were merged back together into Node v4.0.<sup id="cite_ref-41" class="reference"><a href="#cite_note-41">&#91;41&#93;</a></sup> This merge brought V8 <a href="/wiki/ECMAScript#ES2015" title="ECMAScript">ES6</a> features into Node.js and a long-term support release cycle.<sup id="cite_ref-42" class="reference"><a href="#cite_note-42">&#91;42&#93;</a></sup> As of 2016, the io.js website recommends that developers switch back to Node.js and that no further releases of io.js are planned due to the merge.<sup id="cite_ref-43" class="reference"><a href="#cite_note-43">&#91;43&#93;</a></sup>
</p><p>In February 2016, Node.js 0.10.42 is released.<sup id="cite_ref-44" class="reference"><a href="#cite_note-44">&#91;44&#93;</a></sup> 
</p><p>In 2019, the JS Foundation and Node.js Foundation merged to form the <a href="/wiki/OpenJS_Foundation" title="OpenJS Foundation">OpenJS Foundation</a>.
</p>
<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=2" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Node.js allows the creation of <a href="/wiki/Web_server" title="Web server">Web servers</a> and networking tools using <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> and a collection of "modules" that handle various core functionalities.<sup id="cite_ref-b3_28-1" class="reference"><a href="#cite_note-b3-28">&#91;28&#93;</a></sup><sup id="cite_ref-b1_31-2" class="reference"><a href="#cite_note-b1-31">&#91;31&#93;</a></sup><sup id="cite_ref-b2_45-0" class="reference"><a href="#cite_note-b2-45">&#91;45&#93;</a></sup><sup id="cite_ref-b4_46-0" class="reference"><a href="#cite_note-b4-46">&#91;46&#93;</a></sup><sup id="cite_ref-b5_47-0" class="reference"><a href="#cite_note-b5-47">&#91;47&#93;</a></sup> Modules are provided for <a href="/wiki/File_system" title="File system">file system</a> I/O, networking (<a href="/wiki/Domain_Name_System" title="Domain Name System">DNS</a>, <a href="/wiki/HTTP" class="mw-redirect" title="HTTP">HTTP</a>, <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">TCP</a>,  <a href="/wiki/Transport_Layer_Security" title="Transport Layer Security">TLS/SSL</a>, or <a href="/wiki/User_Datagram_Protocol" title="User Datagram Protocol">UDP</a>), <a href="/wiki/Binary_file" title="Binary file">binary</a> data (buffers), <a href="/wiki/Cryptography" title="Cryptography">cryptography</a> functions, <a href="/wiki/Stream_(computing)" title="Stream (computing)">data streams</a>, and other core functions.<sup id="cite_ref-b1_31-3" class="reference"><a href="#cite_note-b1-31">&#91;31&#93;</a></sup><sup id="cite_ref-b4_46-1" class="reference"><a href="#cite_note-b4-46">&#91;46&#93;</a></sup><sup id="cite_ref-48" class="reference"><a href="#cite_note-48">&#91;48&#93;</a></sup> Node.js's modules use an API designed to reduce the complexity of writing server applications.<sup id="cite_ref-b1_31-4" class="reference"><a href="#cite_note-b1-31">&#91;31&#93;</a></sup><sup id="cite_ref-b4_46-2" class="reference"><a href="#cite_note-b4-46">&#91;46&#93;</a></sup>
</p><p>JavaScript is the only language that Node.js supports natively, but many <a href="/wiki/Source-to-source_compiler" title="Source-to-source compiler">compile-to-JS</a> languages are available.<sup id="cite_ref-49" class="reference"><a href="#cite_note-49">&#91;49&#93;</a></sup> As a result, Node.js applications can be written in <a href="/wiki/CoffeeScript" title="CoffeeScript">CoffeeScript</a>,<sup id="cite_ref-b6_50-0" class="reference"><a href="#cite_note-b6-50">&#91;50&#93;</a></sup> <a href="/wiki/Dart_(programming_language)" title="Dart (programming language)">Dart</a>, <a href="/wiki/TypeScript" title="TypeScript">TypeScript</a>, <a href="/wiki/ClojureScript" class="mw-redirect" title="ClojureScript">ClojureScript</a> and others.
</p><p>Node.js is primarily used to build network programs such as Web servers.<sup id="cite_ref-b2_45-1" class="reference"><a href="#cite_note-b2-45">&#91;45&#93;</a></sup> The most significant difference between Node.js and <a href="/wiki/PHP" title="PHP">PHP</a> is that most functions in PHP <a href="/wiki/Asynchronous_I/O" title="Asynchronous I/O">block</a> until completion (commands execute only after previous commands finish), while Node.js functions are <a href="/wiki/Asynchronous_I/O" title="Asynchronous I/O">non-blocking</a> (commands execute <a href="/wiki/Concurrent_computing" title="Concurrent computing">concurrently</a> or even in <a href="/wiki/Parallel_computing" title="Parallel computing">parallel</a>,<sup id="cite_ref-node_cluster_module_doc_51-0" class="reference"><a href="#cite_note-node_cluster_module_doc-51">&#91;51&#93;</a></sup><sup id="cite_ref-52" class="reference"><a href="#cite_note-52">&#91;52&#93;</a></sup> and use <a href="/wiki/Callback_(computer_programming)" title="Callback (computer programming)">callbacks</a> to signal completion or failure).<sup id="cite_ref-b2_45-2" class="reference"><a href="#cite_note-b2-45">&#91;45&#93;</a></sup>
</p><p>Node.js is officially supported on <a href="/wiki/Linux" title="Linux">Linux</a>, <span class="nowrap"><a href="/wiki/MacOS" title="MacOS">macOS</a></span> and <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> 8.1 and Server 2012 (and later),<sup id="cite_ref-supportedOS_3-1" class="reference"><a href="#cite_note-supportedOS-3">&#91;3&#93;</a></sup> with  tier 2 support for <a href="/wiki/SmartOS" title="SmartOS">SmartOS</a> and <a href="/wiki/IBM_AIX" title="IBM AIX">IBM AIX</a> and experimental support for <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a>.  <a href="/wiki/OpenBSD" title="OpenBSD">OpenBSD</a> also works, and LTS versions available for <a href="/wiki/IBM_i" title="IBM i">IBM i</a> (AS/400).<sup id="cite_ref-53" class="reference"><a href="#cite_note-53">&#91;53&#93;</a></sup> The provided source code may also be built on similar operating systems to those officially supported or be modified by third parties to support others such as <a href="/wiki/NonStop_OS" class="mw-redirect" title="NonStop OS">NonStop OS</a><sup id="cite_ref-bomBora_-_Node.js_for_NonStop_54-0" class="reference"><a href="#cite_note-bomBora_-_Node.js_for_NonStop-54">&#91;54&#93;</a></sup> and <a href="/wiki/Unix" title="Unix">Unix</a> servers.
</p>
<h3><span class="mw-headline" id="Platform_architecture">Platform architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=3" title="Edit section: Platform architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Node.js brings <a href="/wiki/Event-driven_programming" title="Event-driven programming">event-driven programming</a> to <a href="/wiki/Web_server" title="Web server">web servers</a>, enabling development of fast web servers in JavaScript.<sup id="cite_ref-b1_31-5" class="reference"><a href="#cite_note-b1-31">&#91;31&#93;</a></sup> Developers can create scalable servers without using <a href="/wiki/Thread_(computing)" title="Thread (computing)">threading</a>, by using a simplified model of <a href="/wiki/Event-driven_programming" title="Event-driven programming">event-driven programming</a> that uses callbacks to signal the completion of a task.<sup id="cite_ref-b1_31-6" class="reference"><a href="#cite_note-b1-31">&#91;31&#93;</a></sup>  Node.js connects the ease of a scripting language (JavaScript) with the power of Unix network programming.<sup id="cite_ref-b1_31-7" class="reference"><a href="#cite_note-b1-31">&#91;31&#93;</a></sup>
</p><p>Node.js was built on top of <a href="/wiki/Google" title="Google">Google</a>'s <a href="/wiki/V8_JavaScript_engine" class="mw-redirect" title="V8 JavaScript engine">V8 JavaScript engine</a> since it was open-sourced under the <a href="/wiki/BSD_license" class="mw-redirect" title="BSD license">BSD license</a>. It is proficient with internet fundamentals such as <a href="/wiki/HTTP" class="mw-redirect" title="HTTP">HTTP</a>, <a href="/wiki/DNS" class="mw-redirect" title="DNS">DNS</a>, and <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">TCP</a>.<sup id="cite_ref-b3_28-2" class="reference"><a href="#cite_note-b3-28">&#91;28&#93;</a></sup> <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> was also a well-known language, making Node.js accessible to the <a href="/wiki/Web_developer" title="Web developer">web development community</a>.<sup id="cite_ref-b3_28-3" class="reference"><a href="#cite_note-b3-28">&#91;28&#93;</a></sup>
</p>
<h3><span class="mw-headline" id="Industry_support">Industry support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=4" title="Edit section: Industry support">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>There are thousands of open-source libraries for Node.js, most of them hosted on the <a href="/wiki/Npm_(software)" title="Npm (software)">npm</a> website. There are multiple developer conferences and events that support the Node.js community, including NodeConf, Node Interactive, and Node Summit as well as a number of regional events.
</p><p>The open-source community has developed <a href="/wiki/Web_framework" title="Web framework">web frameworks</a> to accelerate the development of applications. Such frameworks include Connect, <a href="/wiki/Express.js" title="Express.js">Express.js</a>, <a href="/wiki/Socket.IO" title="Socket.IO">Socket.IO</a>, Feathers.js, Koa.js, Hapi.js, <a href="/wiki/Sails.js" title="Sails.js">Sails.js</a>, <a href="/wiki/Meteor_(web_framework)" title="Meteor (web framework)">Meteor</a>, Derby, and many others.<sup id="cite_ref-b1_31-8" class="reference"><a href="#cite_note-b1-31">&#91;31&#93;</a></sup><sup id="cite_ref-b7_55-0" class="reference"><a href="#cite_note-b7-55">&#91;55&#93;</a></sup> Various packages have also been created for interfacing with other languages or runtime environments such as <a href="/wiki/Microsoft_.NET" class="mw-redirect" title="Microsoft .NET">Microsoft .NET</a>.<sup id="cite_ref-56" class="reference"><a href="#cite_note-56">&#91;56&#93;</a></sup>
</p><p>Modern desktop <a href="/wiki/Integrated_development_environment" title="Integrated development environment">IDEs</a> provide editing and debugging features specifically for Node.js applications. Such IDEs include <a href="/wiki/Atom_(text_editor)" title="Atom (text editor)">Atom</a>, <a href="/wiki/Brackets_(text_editor)" title="Brackets (text editor)">Brackets</a>, <a href="/wiki/JetBrains_MPS" title="JetBrains MPS">JetBrains</a> <a href="/wiki/WebStorm" class="mw-redirect" title="WebStorm">WebStorm</a>,<sup id="cite_ref-57" class="reference"><a href="#cite_note-57">&#91;57&#93;</a></sup><sup id="cite_ref-58" class="reference"><a href="#cite_note-58">&#91;58&#93;</a></sup> <a href="/wiki/Microsoft_Visual_Studio" title="Microsoft Visual Studio">Microsoft Visual Studio</a> (with Node.js Tools for Visual Studio,<sup id="cite_ref-59" class="reference"><a href="#cite_note-59">&#91;59&#93;</a></sup> or <a href="/wiki/TypeScript" title="TypeScript">TypeScript</a> with Node definitions,<sup id="cite_ref-60" class="reference"><a href="#cite_note-60">&#91;60&#93;</a></sup><sup id="cite_ref-61" class="reference"><a href="#cite_note-61">&#91;61&#93;</a></sup><sup id="cite_ref-62" class="reference"><a href="#cite_note-62">&#91;62&#93;</a></sup><sup id="cite_ref-63" class="reference"><a href="#cite_note-63">&#91;63&#93;</a></sup>) <a href="/wiki/NetBeans" title="NetBeans">NetBeans</a>,<sup id="cite_ref-64" class="reference"><a href="#cite_note-64">&#91;64&#93;</a></sup> <a href="/wiki/Nodeclipse" title="Nodeclipse">Nodeclipse</a> Enide Studio<sup id="cite_ref-65" class="reference"><a href="#cite_note-65">&#91;65&#93;</a></sup> (<a href="/wiki/Eclipse_(software)" title="Eclipse (software)">Eclipse</a>-based), and <a href="/wiki/Visual_Studio_Code" title="Visual Studio Code">Visual Studio Code</a>.<sup id="cite_ref-66" class="reference"><a href="#cite_note-66">&#91;66&#93;</a></sup><sup id="cite_ref-67" class="reference"><a href="#cite_note-67">&#91;67&#93;</a></sup>
Certain online web-based IDEs also support Node.js, such as <a href="/wiki/Codeanywhere" title="Codeanywhere">Codeanywhere</a>, <a href="/wiki/Codenvy" class="mw-redirect" title="Codenvy">Codenvy</a>, <a href="/wiki/Cloud9_IDE" title="Cloud9 IDE">Cloud9 IDE</a>, Koding, and the visual flow editor in <a href="/wiki/Node-RED" title="Node-RED">Node-RED</a>.
</p><p>Node.js is supported across a number of cloud-hosting platforms like <a href="/wiki/Jelastic" title="Jelastic">Jelastic</a>, <a href="/wiki/Google_Cloud_Platform" title="Google Cloud Platform">Google Cloud Platform</a>, <a href="/wiki/AWS_Elastic_Beanstalk" title="AWS Elastic Beanstalk">AWS Elastic Beanstalk</a>, <a href="/wiki/Joyent" title="Joyent">Joyent</a> and others.
</p>

<p>New major releases of Node.js are cut from the <a href="/wiki/GitHub" title="GitHub">GitHub</a> master branch every six months. Even-numbered versions are cut in April and odd-numbered versions are cut in October. When a new odd version is released, the previous even version undergoes transition to <a href="/wiki/Long-term_support" title="Long-term support">Long Term Support</a> (LTS), which gives that version 18 months of active support from the date it is designated LTS. After these 18 months expire, an LTS release receives an additional 12 months of maintenance support. An active version receives non-breaking backports of changes a few weeks after they land in the current release. A maintenance release receives only critical fixes and documentation updates.<sup id="cite_ref-Release_68-7" class="reference"><a href="#cite_note-Release-68">&#91;68&#93;</a></sup> The LTS Working Group manages strategy and policy in collaboration with the Technical Steering Committee of the Node.js Foundation.
</p>
<div style="clear:both;"></div>
<h2><span class="mw-headline" id="Technical_details">Technical details</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=6" title="Edit section: Technical details">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Node.js is a JavaScript runtime environment that processes incoming requests in a loop, called the <a href="/wiki/Event_loop" title="Event loop">event loop</a>.
</p>
<h3><span class="mw-headline" id="Internals">Internals</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=7" title="Edit section: Internals">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Node.js uses <a href="/wiki/Libuv" title="Libuv">libuv</a> under the hood to handle asynchronous events. Libuv is an abstraction layer for network and file system functionality on both Windows and <a href="/wiki/POSIX" title="POSIX">POSIX</a>-based systems such as Linux, <a href="/wiki/MacOS" title="MacOS">macOS</a>, OSS on <a href="/wiki/NonStop_(server_computers)" title="NonStop (server computers)">NonStop</a>, and Unix.
</p>
<h3><span class="mw-headline" id="Threading">Threading</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=8" title="Edit section: Threading">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Node.js operates on a <a href="/wiki/Single_threading" class="mw-redirect" title="Single threading">single-thread</a> <a href="/wiki/Event_loop" title="Event loop">event loop</a>, using <a href="/wiki/Non-blocking_I/O" class="mw-redirect" title="Non-blocking I/O">non-blocking I/O</a> calls, allowing it to support tens of thousands of concurrent connections without incurring the cost of thread <a href="/wiki/Context_switch" title="Context switch">context switching</a>.<sup id="cite_ref-69" class="reference"><a href="#cite_note-69">&#91;69&#93;</a></sup> The design of sharing a single thread among all the requests that use the <a href="/wiki/Observer_pattern" title="Observer pattern">observer pattern</a> is intended for building highly concurrent applications, where any function performing I/O must use a <a href="/wiki/Callback_(computer_programming)" title="Callback (computer programming)">callback</a>. To accommodate the single-threaded event loop, Node.js uses the <a href="/wiki/Libuv" title="Libuv">libuv</a> libraryâ€”which, in turn, uses a fixed-sized thread pool that handles some of the non-blocking asynchronous I/O operations.<sup id="cite_ref-readwrite_7-1" class="reference"><a href="#cite_note-readwrite-7">&#91;7&#93;</a></sup>
</p><p>A thread pool handles the execution of parallel tasks in Node.js. The main thread function call posts tasks to the shared task queue, which threads in the thread pool pull and execute. Inherently non-blocking system functions such as networking translate to kernel-side non-blocking sockets, while inherently blocking system functions such as file I/O run in a blocking way on their own threads. When a thread in the thread pool completes a task, it informs the main thread of this, which in turn, wakes up and executes the registered callback.
</p><p>A downside of this single-threaded approach is that Node.js does not allow <a href="/wiki/Vertical_scaling" class="mw-redirect" title="Vertical scaling">vertical scaling</a> by increasing the number of <a href="/wiki/CPU_core" class="mw-redirect" title="CPU core">CPU cores</a> of the machine it is running on without using an additional module, such as cluster,<sup id="cite_ref-node_cluster_module_doc_51-1" class="reference"><a href="#cite_note-node_cluster_module_doc-51">&#91;51&#93;</a></sup> StrongLoop Process Manager,<sup id="cite_ref-70" class="reference"><a href="#cite_note-70">&#91;70&#93;</a></sup> or pm2.<sup id="cite_ref-71" class="reference"><a href="#cite_note-71">&#91;71&#93;</a></sup> However, developers can increase the default number of threads in the libuv thread pool. The server <a href="/wiki/Operating_system" title="Operating system">operating system (OS)</a> is likely to distribute these threads across multiple cores.<sup id="cite_ref-72" class="reference"><a href="#cite_note-72">&#91;72&#93;</a></sup> Another problem is that long-lasting computations and other CPU-bound tasks freeze the entire event-loop until completion.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (March 2017)">citation needed</span></a></i>&#93;</sup>
</p>
<h3><span class="mw-headline" id="V8">V8</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=9" title="Edit section: V8">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Chrome_V8" class="mw-redirect" title="Chrome V8">V8</a> is the JavaScript execution engine which was initially built for <a href="/wiki/Google_Chrome" title="Google Chrome">Google Chrome</a>. It was then open-sourced by Google in 2008. Written in <a href="/wiki/C%2B%2B" title="C++">C++</a>, V8 compiles JavaScript source code to native machine code <a href="/wiki/Just-in-time_compilation" title="Just-in-time compilation">at runtime</a>. <sup id="cite_ref-readwrite_7-2" class="reference"><a href="#cite_note-readwrite-7">&#91;7&#93;</a></sup> As of 2016, it also includes Ignition, a <a href="/wiki/Bytecode_interpreter" class="mw-redirect" title="Bytecode interpreter">bytecode interpreter</a>.
</p>
<h3><span class="mw-headline" id="Package_management">Package management</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=10" title="Edit section: Package management">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Npm_(software)" title="Npm (software)">npm</a> is the pre-installed package manager for the Node.js server platform. It installs Node.js programs from the <a rel="nofollow" class="external text" href="https://www.npmjs.com/">npm registry</a>, organizing the installation and management of third-party Node.js programs. Packages in the npm registry can range from simple helper libraries such as <a href="/wiki/Lodash" title="Lodash">Lodash</a> to task runners such as <a href="/wiki/Grunt_(software)" title="Grunt (software)">Grunt</a>.
</p>
<h3><span class="mw-headline" id="Unified_API">Unified API</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=11" title="Edit section: Unified API">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Node.js can be combined with a browser with your site , a database that supports JSON data (such as <a href="/wiki/PostgreSQL" title="PostgreSQL">Postgres</a>,<sup id="cite_ref-73" class="reference"><a href="#cite_note-73">&#91;73&#93;</a></sup> <a href="/wiki/MongoDB" title="MongoDB">MongoDB</a>, or <a href="/wiki/CouchDB" class="mw-redirect" title="CouchDB">CouchDB</a>) and <a href="/wiki/JSON" title="JSON">JSON</a> for a unified JavaScript development stack. With the adaptation of what were essentially server-side development patterns such as <a href="/wiki/Model%E2%80%93view%E2%80%93controller" title="Modelâ€“viewâ€“controller">MVC</a>, <a href="/wiki/Model_View_Presenter" class="mw-redirect" title="Model View Presenter">MVP</a>, <a href="/wiki/Model_View_ViewModel" class="mw-redirect" title="Model View ViewModel">MVVM</a>, etc., Node.js allows the <a href="/wiki/Code_reuse" title="Code reuse">reuse</a> of the same model and service interface between client side and server side.
</p>
<h3><span class="mw-headline" id="Event_loop">Event loop</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=12" title="Edit section: Event loop">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Node.js registers with the operating system so the OS notifies it of connections and issues a callback. Within the Node.js runtime, each connection is a small <a href="/wiki/Heap_allocation" class="mw-redirect" title="Heap allocation">heap allocation</a>. Traditionally, relatively heavyweight OS processes or threads handled each connection. Node.js uses an event loop for scalability, instead of processes or threads.<sup id="cite_ref-74" class="reference"><a href="#cite_note-74">&#91;74&#93;</a></sup> In contrast to other event-driven servers, Node.js's event loop does not need to be called explicitly. Instead, callbacks are defined, and the server automatically enters the event loop at the end of the callback definition. Node.js exits the event loop when there are no further callbacks to be performed.
</p>
<h3><span class="mw-headline" id="WebAssembly">WebAssembly</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=13" title="Edit section: WebAssembly">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Node.js supports <a href="/wiki/WebAssembly" title="WebAssembly">WebAssembly</a> and as of Node 14 has experimental support of <a href="/wiki/WebAssembly#WASI" title="WebAssembly">WASI</a>, the WebAssembly System Interface.
</p>
<h3><span class="mw-headline" id="Native_bindings">Native bindings</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=14" title="Edit section: Native bindings">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Node.js provides a way to make "addons" via a <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a>-based API called N-API which can be used to produce loadable (importable) <code>.node</code> modules from source code written in C/C++.<sup id="cite_ref-:1_75-0" class="reference"><a href="#cite_note-:1-75">&#91;75&#93;</a></sup> The modules can be directly loaded into memory and executed from within JS environment as simple CommonJS modules. The implementation of the N-API relies on internal C/C++ Node.js and V8 objects requiring users <a href="/wiki/Language_binding" title="Language binding">to import</a> (<a href="/wiki/Include_directive" title="Include directive">#include</a>) Node.js specific <a href="/wiki/Include_directive" title="Include directive">headers</a> into their native source code.<sup id="cite_ref-:1_75-1" class="reference"><a href="#cite_note-:1-75">&#91;75&#93;</a></sup> As Node.js platform constantly evolves the API compatibility is subject to changes and may get broken sometimes by a new version (as consequence modules have to be built against specific Node.js versions to work correctly). To address the issue third parties have introduced open-sourced Ð¡/Ð¡++ wrappers on top of the API that partially alleviate the problem. They simplify interfaces but as side effect they may also introduce complexity which maintainers have to deal with. Even though the core functionality of Node.js resides in a JavaScript built-in library, modules written in C++ can be used to enhance capabilities and to improve performance of applications.
</p><p>In order to produce such modules one needs to have an appropriate C++ compiler and necessary headers (the latter are typically shipped with Node.js itself): <a href="/wiki/GCC_Compiler" class="mw-redirect" title="GCC Compiler">gcc</a>, <a href="/wiki/Clang" title="Clang">clang</a> or <a href="/wiki/Microsoft_Visual_C%2B%2B" title="Microsoft Visual C++">MSVC++</a>.
</p><p>The N-API is similar to <a href="/wiki/Java_Native_Interface" title="Java Native Interface">Java Native Interface</a>.
</p>
<h2><span class="mw-headline" id="Project_governance">Project governance</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Node.js&amp;action=edit&amp;section=15" title="Edit section: Project governance">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In 2015, various branches of the greater Node.js community began working under the vendor-neutral Node.js Foundation. The stated purpose of the organization "is to enable widespread adoption and help accelerate development of Node.js and other related modules through an open governance model that encourages participation, technical contribution, and a framework for long-term stewardship by an ecosystem invested in Node.js' success."<sup id="cite_ref-76" class="reference"><a href="#cite_note-76">&#91;76&#93;</a></sup>
</p><p>The Node.js Foundation Technical Steering Committee (TSC) is the technical governing body of the Node.js Foundation. The TSC is responsible for the core Node.js repo as well as dependent and adjacent projects. Generally the TSC delegates the administration of these projects to working groups or committees.<sup id="cite_ref-77" class="reference"><a href="#cite_note-77">&#91;77&#93;</a></sup> The LTS group that manages long term supported releases is one such group. Other current groups include Website, Streams, Build, Diagnostics, i18n, Evangelism, Docker, Addon API, Benchmarking, Post-mortem, Intl, Documentation, and Testing.<sup id="cite_ref-78" class="reference"><a href="#cite_note-78">&#91;78&#93;</a></sup>
</p><p>In August 2017, a third of the TSC members resigned due to a dispute related to the project's code of conduct.<sup id="cite_ref-79" class="reference"><a href="#cite_note-79">&#91;79&#93;</a></sup>
</p>
